## Проблема локального парсинга

Когда вы парсите локально (со своего компьютера):

1. **Свет выключили** — парсинг остановился, задачи потерялись
2. **Интернет пропал** — всё встало
3. **Компьютер занят** — нельзя выключить или перезагрузить
4. **Нужно держать включенным 24/7** — шум, электричество, износ

Плюсы локального парсинга:
- Полный контроль
- Быстрая отладка
- Не нужно загружать файлы на хостинг

Минусы:
- Зависимость от света и интернета
- Нужно держать ПК включенным
- Сложно масштабировать

---

## Проблема парсинга на хостинге

Когда вы парсите с хостинга (Beget):

Плюсы:
- Работает 24/7 независимо от вашего интернета и света
- Не шумит дома
- Электричество не оплачиваете
- Автоматический запуск по расписанию

Минусы:
- Ограничения хостинга (время выполнения, память)
- Все запросы идут с IP хостинга (могут забанить)
- Нужно загружать файлы на сервер
- Отладка сложнее

---

## Главное препятствие: IP адрес хостинга

**Ключевая проблема:**

Если вы разместите парсер на Beget и будете делать запросы напрямую — все запросы будут идти с одного IP адреса (IP серверов Beget).

Китайские и корейские автосайты быстро обнаружат:
- Много запросов с одного IP
- Заблокируют этот IP
- Ваш парсинг остановится

**Вывод: Без прокси НЕЛЬЗЯ парсить с хостинга!**

---

## Решение: Свои прокси на тех же поддоменах Beget

Вот гениальная схема:

### Шаг 1: Создаёте 20-50 поддоменов на Beget

Пример:
- proxy1.yoursite.ru
- proxy2.yoursite.ru
- proxy3.yoursite.ru
- ...
- proxy50.yoursite.ru

**Важно:** Каждый поддомен получает свой уникальный IP адрес!

Beget распределяет поддомены по разным серверам, поэтому IP будут разными.

### Шаг 2: На каждый поддомен ставите PHP proxy скрипт

Это простой скрипт на 100 строк кода. Он принимает запрос от вашего парсера и перенаправляет его на целевой сайт (che168, dongchedi, encar).

Китайский сайт видит:
- Запрос пришёл с IP proxy7.yoursite.ru
- Это обычный пользователь
- Нет признаков парсинга

### Шаг 3: Парсер на Beget использует пул прокси

Ваш основной парсер (тоже на Beget) делает так:
1. Берёт список из 50 прокси
2. Для каждого запроса выбирает случайный прокси
3. Запрос идёт: Парсер → Прокси → Китайский сайт
4. Ответ возвращается обратно

**Результат:**
- Китайские сайты видят запросы с 50 разных IP
- Нет массовых запросов с одного адреса
- Бана не будет

---

## Почему это работает

### Беget распределяет поддомены по разным IP

Когда вы создаёте поддомен на бесплатном хостинге Beget, он размещается на одном из множества серверов компании. У каждого сервера свой IP адрес.

Если создать 50 поддоменов — они распределятся по ~20-50 разным IP (некоторые поддомены могут быть на одном сервере).

### Китайские сайты не видят связи

Для che168.com:
- Запрос с IP 185.244.52.10 — выглядит как обычный пользователь
- Запрос с IP 185.244.52.45 — тоже обычный пользователь
- Запрос с IP 185.244.52.78 — ещё один пользователь

Они не видят, что все эти IP принадлежат поддоменам одного человека.

### Прокси обновляются

Если какой-то IP всё же забанят:
- Отключаете этот поддомен из пула
- Используете остальные 49
- По желанию создаёте новый поддомен

│  che168.com                     │

**Как работает:**

1. Парсер запускается по расписанию (cron)
2. Берёт следующий URL для парсинга
3. Выбирает случайный прокси из пула (или по кругу)
4. Отправляет запрос через прокси
5. Получает HTML страницы
6. Извлекает данные об автомобилях
7. Сохраняет в базу данных (SQLite на том же Beget)
8. Повторяет для следующего URL

## Ответы на важные вопросы

### Вопрос: Не будет ли бана, если все прокси на одном хостинге?

**Ответ:** Не будет, если правильно сделать.

Почему:
1. Beget распределяет поддомены по разным серверам → разные IP
2. 50 поддоменов = примерно 20-50 разных IP адресов
3. Запросы распределяются равномерно (не все на один IP)
4. Китайские сайты не видят связи между поддоменами

**Но важно:**
- Не делать больше 100-200 запросов в час с одного прокси
- Использовать задержки между запросами (2-5 секунд)
- Менять User-Agent для разных запросов
- Если прокси забанили — отключить его из пула

### Вопрос: Сколько прокси нужно?

**Ответ:** Минимум 20, лучше 50, идеально 100.

Расчёт:
- 3 сайта (che168, dongchedi, encar)
- На каждый сайт нужно 7-15 прокси
- Итого 21-45 прокси
- Плюс резерв 10-20%

**Рекомендация:** Создать 50 поддоменов сразу. Если мало — добавить ещё.

### Вопрос: Как часто будут банить прокси?

**Ответ:** Зависит от аккуратности парсинга.

При правильной настройке:
- 1-2 прокси в неделю (некритично)
- При агрессивном парсинге: 5-10 в неделю

**Решение:**
- Иметь запас (50+ прокси)
- Отслеживать failed запросы
- Автоматически исключать забаненные прокси из пула
- Периодически создавать новые поддомены

### Вопрос: Хватит ли ресурсов Beget?

**Ответ:** Хватит с запасом.

Парсинг — это в основном HTTP запросы и простая обработка HTML.

Потребление:
- CPU: 10-20% во время парсинга
- RAM: 200-500 MB
- Диск: 1-2 GB (база + логи)

Тариф Blog (420₽/мес) или Start (590₽/мес) хватит с запасом.

### Вопрос: Что если Beget заблокирует за парсинг?

**Ответ:** Маловероятно, но нужно соблюдать правила.

Правила Beget:
- Не превышать нагрузку на CPU (80% в тарифе)
- Не делать DDoS атаки (у вас не тот случай)
- Не нарушать законы РФ

Парсинг общедоступных сайтов — не нарушение.

**Но для надёжности:**
- Иметь резервный хостинг (Timeweb, Reg.ru)
- Делать бэкапы базы данных
- При блокировке — переехать за 1-2 дня

### Вопрос: Нужен ли отдельный сервер для базы данных?

**Ответ:** Нет, SQLite справится.

SQLite на Beget:
- Файл autos.db размером до 10 GB (хватит на миллионы записей)
- Скорость: тысячи запросов в секунду
- Надёжность: проверено годами

**Когда понадобится PostgreSQL:**
- Если будет 10+ миллионов записей
- Если нужно несколько серверов读写
- Если нужна сложная аналитика

Для вашей задачи — SQLite более чем достаточно.

## Что делать если...

### ...прокси забанили

**Признаки:**
- Ошибка 403 Forbidden
- CAPTCHA вместо страницы
- Таймаут соединения

**Решение:**
1. Пометить прокси как "failed" в конфиге
2. Парсер автоматически исключит его из пула
3. Продолжить работу с остальными прокси
4. По желанию — создать новый поддомен взамен

### ...Beget тормозит

**Признаки:**
- Скрипт выполняется дольше обычного
- Ошибки "Resource limit reached"

**Решение:**
1. Проверить нагрузку в панели Beget
2. Уменьшить частоту парсинга (не каждые 4 часа, а каждые 6)
3. Уменьшить количество одновременных запросов
4. Если постоянно — перейти на тариф выше

### ...закончилось место на диске

**Признаки:**
- Ошибка при записи в базу
- Ошибка при создании логов

**Решение:**
1. Очистить старые логи (папка logs/)
2. Удалить старые бэкапы (оставить за последние 7 дней)
3. Проверить размер базы данных
4. Если нужно — купить больше места (на Beget недорого)

### ...китайский сайт изменил структуру

**Признаки:**
- Парсер не находит данные на странице
- В логе ошибки парсинга

**Решение:**
1. Открыть страницу вручную, посмотреть HTML
2. Найти новые селекторы (классы, ID)
3. Обновить парсер (che168.py, etc.)
4. Протестировать на 1-2 страницах
5. Запустить заново

## Безопасность и скрытность

### Как не попасться

**Делать:**
- Задержки между запросами (2-5 секунд)
- Разные User-Agent для разных запросов
- Распределение нагрузки (не все запросы сразу)
- Использовать много прокси (50+)
- Мониторить failed запросы

**Не делать:**
- Не парсить чаще 1 раза в 4 часа
- Не делать больше 1000 запросов в час
- Не игнорировать ошибки (403, CAPTCHA)
- Не парсить все страницы подряд (только новые)

### Токены безопасности на прокси

В proxy.php добавить проверку токена:

```php
$token = $_GET['token'] ?? '';
if ($token !== 'ваш_секретный_токен') {
    die('Unauthorized');
}
```

В парсере передавать токен с каждым запросом. Это защитит от использования ваших прокси другими людьми.
